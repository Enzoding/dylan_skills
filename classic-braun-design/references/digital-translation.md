# Digital Translation

Use this reference when applying classic Braun product language to websites, apps, and other coding-agent outputs. These are examples and prompts, not mandatory templates.

## Adaptation Prompts

Before designing, decide:
- Should this interface feel like a device face, control panel, modular component system, clock dial, calculator, catalog page, or instrument readout?
- Which 2-4 Braun traits naturally support this task?
- Which tempting traits should be omitted so the result stays clean?
- What must remain modern for usability: navigation, touch targets, responsive layout, accessibility, forms, and states?

## Page Structure

Translate physical product structure into digital structure:
- Product casing -> page shell, app frame, or main surface.
- Faceplate -> primary content area.
- Control strip -> toolbar, nav rail, bottom bar, command area.
- Display window -> hero readout, data panel, preview, chart, or current state.
- Equipment label -> eyebrow, metadata, route label, caption, status label.
- Module stack -> sections, cards, settings groups, dashboard panels.
- Grille or perforation field -> texture, loading surface, background detail, or audio/weather/status motif.

Use these only when they serve the interface. Do not force a device metaphor onto content that needs simple reading.

## Webpages

For webpages:
- Treat the first viewport as a product face: clear identity, primary state, main action, and a controlled visual system.
- Avoid marketing-page defaults: oversized gradient hero, centered generic copy, floating card pile, soft blur background.
- Use modular sections with precise labels and consistent alignment.
- Let product-style details appear in navigation, specs, controls, dividers, and feature modules.
- Use small accent color for action or status, not broad decoration.

## Apps

For apps:
- Keep repeated-use surfaces calm and efficient.
- Use device-like control groups for settings, filters, playback, generation controls, or mode switching.
- Preserve modern affordances: touch targets, focus states, keyboard support, readable labels, and responsive behavior.
- Use tabular numeric displays and measured scales for metrics and states.
- Avoid making every screen a literal machine panel; reserve strong device framing for key views.

## Dashboards and Tools

For dashboards and operational tools:
- Use module stacks, instrument panels, status lights, segmented filters, and fine dividers.
- Make hierarchy denser but highly ordered.
- Group readouts like audio equipment displays or control surfaces.
- Use accents for alert, active, selected, live, or power states.
- Make tables feel like precision readouts, not generic enterprise grids.

## Forms and Settings

For forms:
- Use label proximity and strict grouping.
- Treat fieldsets like labeled equipment sections.
- Use low-radius inputs, clear focus rings, and explicit validation states.
- Use toggle and segmented controls with physical clarity.
- Do not make forms too small or cryptic for the sake of product styling.

## Mobile

For mobile:
- Use compact product-face logic: display area plus control area.
- Keep tap targets modern and comfortable.
- Use bottom controls only when they fit the task.
- Use circular controls and calculator-like buttons sparingly.
- Avoid dense tiny labels that fail on small screens.

## Responsiveness

- Preserve the grid by changing module count, not by letting panels collapse randomly.
- Keep labels attached to their controls.
- Avoid tiny technical UI on mobile.
- Let ornamental details disappear before core information.
- Make product framing adapt: full faceplate on desktop, compact device surface on mobile.

## Interaction

- Make buttons feel pressable through border, fill, inset contrast, or clear affordance.
- Use subtle state changes: active lamps, tick movement, depressed key, highlighted segment, changed readout.
- Use motion for functional continuity, not spectacle.
- Keep transitions quick and mechanical rather than elastic or magical.

## Copy

- Prefer precise product language over hype.
- Use labels, captions, measurements, states, and concise commands.
- Avoid breathless marketing phrases.
- Make limitations and statuses visible.

## Common Failure Modes

- The page becomes merely grey and empty.
- The style becomes generic retro rather than Braun product language.
- Controls look decorative but are interactive.
- Accent colors are used as decoration rather than function.
- The grid is strict but the content hierarchy is weak.
- Text becomes too small to read.
- The interface sacrifices modern usability to imitate a physical object.
